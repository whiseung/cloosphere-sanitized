# Changelog 작성

버전별 변경 로그를 `CHANGELOG.ko.md`와 `CHANGELOG.md`에 한/영 동시 작성합니다.

## 버전 입력

**버전 번호는 필수입니다.** 사용자가 버전을 제공하지 않으면, 반드시 버전 번호를 입력해달라고 요청하세요.

예시: `/changelog 0.7.6` 또는 `/changelog` → "버전 번호를 입력해주세요 (예: 0.7.6)"

## 작성 원칙

### 대상 독자: 고객 (관리자/사용자)

변경 로그는 **코드 레벨 설명이 아닌, 기능의 특장점과 이점**을 설명해야 합니다.

| 잘못된 예 (코드 레벨) | 올바른 예 (특장점) |
|---|---|
| `run_sql` 도구의 description에서 `table_overview`, `memory_context` 파라미터를 제거 | 필요한 정보만 선별적으로 가져오므로 토큰 사용량이 크게 감소하고 응답 속도가 향상됩니다 |
| OData 서버 필터를 사용하여 클라이언트 필터링 제거 | 대규모 데이터베이스에서도 빠르게 필요한 테이블 정보를 가져옵니다 |
| `ToolConnectionManager` 모듈 추가 | 다수의 도구가 연결되어 있어도 불필요한 호출 없이 효율적으로 동작합니다 |

### 작성 톤

- 기능이 **어떤 형태로 개선되어 어떤 이점이 생겼는지** 중심으로 서술
- 코드명, 함수명, 변수명 등 기술 용어 사용 금지
- 사용자가 체감할 수 있는 변화를 강조

### 간결성 원칙 (중요)

CHANGELOG는 **짧고 굵게**. 커밋 로그를 나열하는 문서가 아닙니다.

**분량 목표**: 메이저/마이너 릴리즈 30~50줄, 패치 릴리즈 15~30줄 이내 (섹션 당 항목 3~7개)

**포함 기준** — 사용자가 **"무엇이 달라졌어?"** 라고 물었을 때 답이 되는 내용만:
- ✓ 신규 대분류 기능 (예: "Teams 봇 통합", "KG Phase 2")
- ✓ 사용자 체감이 큰 UX·성능 개선
- ✓ UI 구조 변경 (새 탭, 새 메뉴, 새 설정 화면)
- ✓ 데이터 마이그레이션이 수반되는 변경
- ✓ 권한·보안 정책 변화

**제외 기준** — 내부 개선·세세한 버그는 **넣지 않거나 통합**합니다:
- ✗ 내부 리팩토링 (구조 변경 없음)
- ✗ 개별 필드/버튼/모달 세부 버그
- ✗ 로케일·번역 자잘한 수정
- ✗ 코드 스타일/린트/포맷
- ✗ 테스트·CI 변경
- ✗ "…되던 버그 수정" 류 단발성 fix 수십 개 나열

**통합 패턴** — 여러 세부 fix/개선은 한 줄로 묶습니다:

| 넣지 말 것 | 대신 이렇게 통합 |
|---|---|
| 긴 사용자 이름 말줄임 / 사이드바 새로고침 / 툴팁 위치 / ... 개별 5~10개 나열 | **🔧 UI 미세 개선**: 긴 이름 말줄임, 새로고침 동기화 등 자잘한 UI 버그가 정리되었습니다. |
| Redis/alembic/멀티인스턴스/마이그레이션 등 각각 | **🔧 안정성·호환성 개선**: 외부 의존성 실패 시 빠른 감지, 커스텀 스키마 호환, 멀티 인스턴스 환경 안전성이 강화되었습니다. |
| 번역 키 추가 10개 / 로케일 parity 수정 / ... | **🌐 다국어 보강**: 신규 UI 문자열 번역과 로케일 정합성 보강. |

**분류할 때 항상 물어볼 것**: "이걸 고객이 알면 뭐가 달라지는가?" — 답이 없으면 제외합니다.

## 작업 순서

### 1단계: 변경 범위 파악

이전 버전 CHANGELOG의 마지막 날짜 이후 커밋을 분석합니다.

```bash
# 기존 CHANGELOG에서 직전 버전 확인
head -20 CHANGELOG.ko.md

# 직전 버전 CHANGELOG 커밋 이후의 변경 확인
git log <직전_changelog_커밋>..HEAD --oneline --no-merges | grep -vE "docs:|style:|chore:"
```

주요 커밋의 상세 메시지도 확인합니다:
```bash
git log -1 --format="%B" <커밋해시>
```

### 2단계: 분류

커밋을 Keep a Changelog 카테고리로 분류합니다:

| 카테고리 | 대상 |
|----------|------|
| **Added** | 새 기능, 새 옵션 |
| **Fixed** | 버그 수정, 보안 수정 |
| **Changed** | 기존 기능 개선, 성능 향상, UX 변경 |

### 3단계: 작성

`CHANGELOG.ko.md`와 `CHANGELOG.md` 모두 업데이트합니다.

**형식:**
```markdown
## [버전] - 연도-월

### Added

- **이모지 제목**: 한 줄 설명. 추가 상세 설명.

### Fixed

- **이모지 제목**: 어떤 문제가 어떻게 수정되었는지.

### Changed

- **이모지 제목**: 무엇이 어떻게 개선되었는지.
```

**이모지 컨벤션** (기존 CHANGELOG 참고):
- 📊 데이터/분석/DbSphere
- 🔧 설정/도구/개선
- 💻 Code Gateway
- 📈 모니터링
- 💬 채팅/메시징
- 🔍 검색/지식기반
- 🛡️ 가드레일/보안
- 📅 스케줄/작업
- 📝 문서/텍스트
- 🔐 권한/라이선스

### 4단계: 검증

- 기존 버전 항목과 톤, 형식 일관성 확인
- 한/영 내용 대응 확인
- 코드 레벨 용어가 포함되지 않았는지 최종 확인

## 주의사항

- **커밋하지 않음**: 작성 후 사용자 검토를 기다립니다
- `docs:`, `style:`, `chore:` 타입 커밋은 일반적으로 CHANGELOG에 포함하지 않습니다
- 이전 버전 `## [x.x.x]` 블록 바로 위에 새 버전을 삽입합니다
