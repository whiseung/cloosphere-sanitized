# 릴리즈 문서 일괄 업데이트 (CHANGELOG + 가이드)

버전 릴리즈에 필요한 **CHANGELOG와 사용자 가이드**를 한/영 동시에 일괄 업데이트합니다.

개별 작업은 `/changelog`, `/guide_update`로도 가능하지만, 릴리즈 시점에는 본 통합 command로 한 번에 처리하세요.

## 사용법

```
/release_docs <버전>
```

- 버전 번호는 필수. 누락 시 입력 요청.
- 예: `/release_docs 1.0.1`
- **버전과 git tag 의 관계는 무시**. 사용자가 명시한 버전을 그대로 사용한다. tag 가 이미 cut 됐든(예: `v1.0.3` 이 이미 존재) 안 됐든, 중간 버전(`v1.0.2`) 의 changelog 가 빠져있든 상관없이 — "마지막으로 가이드/CHANGELOG 가 작성된 시점부터 HEAD 까지의 변경 전체" 를 사용자가 지정한 버전 한 항목으로 묶는다. 누락된 중간 patch 버전을 "쪼개서 기록할까요?" 같은 질문은 **하지 않는다** (사용자 가시 가치 0).

## 두 문서의 역할 분담

| 문서 | 목적 | 톤 | 분량 |
|------|------|----|----|
| **CHANGELOG** | "이 버전에서 무엇이 달라졌나" 한눈에 | 짧고 굵게 | 패치 15~30줄, 마이너 30~50줄 |
| **Guide** | "새 기능을 어떻게 쓰나" 따라할 수 있게 | 풍부하고 정확하게 | 제한 없음 |

**핵심 원칙**: CHANGELOG는 *요약*, 가이드는 *사용법*. 세부 절차를 CHANGELOG에 쓰지 말고 가이드로 링크/인도하세요.

## 작업 순서

### Phase 0 — 변경 범위 파악

#### 0-1. baseline 식별 (절대 git tag 로 잡지 말 것)

baseline 은 **마지막으로 CHANGELOG 또는 guide/ 가 갱신된 commit**. git tag 는 사람이 따로 cut 하므로 changelog/guide 작성 시점과 어긋나기 쉽다 (예: `v1.0.2`, `v1.0.3` tag 만 있고 changelog 는 `v1.0.1` 까지만 — 그 사이 작업은 *어디에도* 기록 안 됨).

```bash
# baseline = CHANGELOG 또는 guide 가 마지막으로 변경된 commit
BASELINE=$(git log -1 --format="%h" -- CHANGELOG.md guide/)
echo "baseline = $BASELINE ($(git log -1 --format='%ai %s' $BASELINE))"
```

이 명령 한 번이면 충분. 만약 사용자가 "이전에 v1.0.x 까지는 기록됐을 텐데" 하더라도 **이 BASELINE 을 신뢰**한다 — 누락된 중간 commit 들은 모두 이번 버전으로 흡수.

#### 0-2. 범위 계산

```bash
# baseline..HEAD (현재 브랜치 끝까지). docs/style/chore 제외
git log $BASELINE..HEAD --oneline --no-merges | grep -vE "docs:|style:|chore:" > /tmp/release_commits.txt
wc -l /tmp/release_commits.txt
cat /tmp/release_commits.txt

# UI / 가이드 영향 식별
git diff $BASELINE..HEAD --stat -- src/lib/components/ src/routes/ guide/ | head -80
```

#### 0-3. 분량 추정 (commit 수 → 릴리즈 유형)

| baseline..HEAD commit 수 | 권장 릴리즈 유형 | CHANGELOG 분량 |
|---|---|---|
| ~20 | 패치 | 15~30줄 |
| 20~80 | 마이너 | 30~50줄 |
| 80+ | 메이저 또는 두 번에 걸친 누적 — 마이너 분량 (50줄) 으로 묶고 핵심만 | 50~80줄 |

사용자가 지정한 버전이 patch 형식 (x.x.+1) 이라도 **commit 수가 마이너 수준이면 마이너 분량으로 작성**한다 — 분량은 "변경 영향" 기준이지 "버전 번호 형식" 기준이 아님.

#### 0-4. 작업 범위가 너무 넓을 때

100+ commit / 30+ UI 컴포넌트 변경이면 **카테고리별로 grep + 파일별 stat** 으로 좁혀서 파악:

```bash
# 카테고리별 키워드로 묶음 식별
grep -iE "kms|encrypt|key vault" /tmp/release_commits.txt
grep -iE "kg |knowledge.graph|cypher|kg_" /tmp/release_commits.txt
grep -iE "embed|widget" /tmp/release_commits.txt
grep -iE "teams|notification|webhook" /tmp/release_commits.txt
```

### Phase 1 — 분류

커밋을 **사용자 가시 여부**와 **기능 단위**로 분류합니다.

분류를 끝내고 사용자에게 한 번 보고 후 진행 — 단 묻는 건 **"이 분류로 진행해도 되나"** 한 줄. baseline / 누락 patch 버전 / 분량 같은 것을 다시 묻지 말 것 (Phase 0 에서 결정된 사실).

#### 1-A. 대분류 기능 (CHANGELOG `Added` + Guide 대규모 추가)
- 완전 신규 기능 모듈 (예: Teams 봇, BigQuery 커넥터)
- 기존 기능 대규모 확장 (예: 임베드 위젯에 게스트/SSO/새 모드 추가)
- 새 관리자 탭·페이지 추가
- 명확히 "사용자가 알아야 할" 기능

→ **CHANGELOG 주요 불릿으로 기재 + Guide 해당 섹션 대폭 업데이트**

#### 1-B. UX/성능 개선 (CHANGELOG `Changed` + Guide 보강)
- 화면 구성 통일
- 성능 최적화 (눈에 띄는 속도 향상)
- 권한/정책 변경

→ **CHANGELOG Changed에 한 줄 + Guide 관련 부분 보강**

#### 1-C. 작은 버그 수정·안정화 (CHANGELOG 통합 or 제외)
- 개별 필드/모달/UI 세부 버그
- 의존성 호환성
- 내부 에러 로깅 개선

→ **CHANGELOG에는 통합 Fixed 1~2줄로 요약 또는 제외** (개별 나열 금지)

#### 1-D. 내부 리팩토링·CI·테스트
- 코드 구조 개선 (동작 변경 없음)
- 테스트/CI/빌드 설정

→ **CHANGELOG 제외**. Guide 영향 없으면 스킵.

#### UI 변경 체크리스트 (Guide 동기화 필수)

다음 중 하나라도 해당하면 **가이드 반드시 업데이트**:
- [ ] 새 탭·새 페이지·새 모달 (`src/lib/components/**/*.svelte` 신규 파일)
- [ ] 설정 항목 추가/이름 변경 (Settings 하위 변경)
- [ ] 버튼·메뉴 위치/명칭 변경
- [ ] 폼 필드 추가·제거
- [ ] 권한 레벨·역할 체계 변화
- [ ] 지원 대상 추가 (DB, LLM 프로바이더, OAuth 등)
- [ ] 사이드바 메뉴 변경 (`_sidebar.md` 영향)

기준: *"사용자가 가이드만 보고 실제 화면을 찾을 수 있는가?"*

#### "운영자 가시" 도 사용자 가시로 취급

대규모 신기능이 관리자 전용 (KMS, 자동 회전, 감사 로그 등) 이라도 가이드 의 `admin/` 영역에 풀 섹션 / 신규 파일을 만들어야 한다. "최종 사용자가 안 보니 빼자" 는 NO — 기능을 켜야 할 운영자가 가이드 사용자다.

#### 신규 가이드 파일 = 사이드바 등록 필수

`guide/{ko,en}/admin/encryption.md` 같은 신규 파일을 만들었다면 같은 PR 에서 `guide/{ko,en}/_sidebar.md` 의 해당 섹션 (보통 `**관리자**` 그룹) 에 한/영 모두 등록해야 한다. 등록 안 하면 docsify 가 못 찾아 죽은 페이지가 됨.

### Phase 2 — 가이드 먼저 업데이트

CHANGELOG 전에 **가이드를 먼저** 씁니다. 이유: 
- 가이드 작업 중 "이 기능을 어떻게 한 줄로 요약할까?" 고민이 자연스럽게 CHANGELOG 문구가 됨
- 가이드에 못 쓸 만큼 모호하면 CHANGELOG에도 넣지 말아야 함

**병렬 Agent 활용**: 범위가 넓으면 카테고리별로 Agent 2~3개 병렬 실행.
- Agent A: 관리자 영역 (`admin/*.md`)
- Agent B: 워크스페이스 영역 (`workspace/*.md`)
- Agent C: 루트 및 공통 (`chat.md`, `schedules.md` 등)

각 Agent에게 전달할 필수 정보:
1. 담당 파일 목록 (한/영 짝)
2. 반영할 신기능 목록 (Phase 1 분류 기반)
3. 실제 UI 라벨을 코드에서 확인하도록 지시 (추측 금지)
4. 기존 톤·형식 유지, 한/영 1:1 대응
5. 누락된 부분만 추가, 기존 내용 삭제 금지

가이드 원칙은 `/guide_update`와 동일 — 해당 command 본문 참조.

### Phase 3 — CHANGELOG 작성 (간결하게)

가이드 작업이 완료되면, 가이드에서 추가된 섹션 제목들을 기반으로 CHANGELOG를 **짧게** 작성합니다.

#### 분량 목표 (엄수)

| 릴리즈 유형 | 전체 줄 수 | Added 항목 | Changed 항목 | Fixed 항목 |
|---|---|---|---|---|
| 패치 (x.x.+1) | **15~30줄** | 3~5 | 1~3 | 1~3 (통합) |
| 마이너 (x.+1.0) | **30~50줄** | 5~8 | 2~4 | 2~4 (통합) |
| 메이저 (+1.0.0) | 50~80줄 | 7~12 | 3~5 | 3~5 |

#### 항목 당 형식

```markdown
- **🎯 기능명**: 한 문장 요약. 핵심 이점 한 문장 추가 가능.
```

- 불릿 하나 당 **2문장 이내**
- 하위 불릿은 **정말 필요할 때만** (신규 대분류 기능은 2~3개까지 허용)
- 세부 사양·절차는 가이드로 미루기

#### 포함 vs 제외

**포함** (고객이 바로 체감하는 것):
- ✓ 신규 대분류 기능
- ✓ UX·성능 개선 (체감 크기)
- ✓ UI 구조 변경
- ✓ 데이터 마이그레이션 수반 변경
- ✓ 권한·보안 정책 변화

**제외** (통합하거나 생략):
- ✗ 내부 리팩토링
- ✗ 개별 필드/버튼/모달 세부 버그
- ✗ 로케일 자잘한 수정
- ✗ 코드 스타일/린트/포맷
- ✗ 테스트·CI
- ✗ "…되던 버그 수정" 단발성 fix 나열

#### 통합 패턴 (여러 fix를 한 줄로)

| 넣지 말 것 | 대신 이렇게 |
|---|---|
| 긴 이름 말줄임 / 사이드바 새로고침 / 툴팁 위치 등 5~10개 나열 | **🔧 UI 미세 개선**: 자잘한 UI 버그가 정리되었습니다. |
| Redis 재시도 / alembic 호환 / 멀티인스턴스 / ... | **🔧 안정성·호환성 개선**: 외부 의존성 감지, 커스텀 스키마 호환, 멀티 인스턴스 안전성이 강화되었습니다. |
| 번역 키 추가 / 로케일 parity / ... | **🌐 다국어 보강** 한 줄로 통합 |

**기준**: "이걸 고객이 알면 뭐가 달라지는가?" 답이 없으면 제외.

### Phase 4 — 검증

1. **한/영 1:1 대응**
   ```bash
   # 섹션 수 비교
   grep -c "^### " CHANGELOG.md CHANGELOG.ko.md
   grep -c "^### " guide/en/**/*.md guide/ko/**/*.md | paste - -
   ```

2. **이전 버전 톤 일관성**: 기존 `[1.0.0]`, `[0.8.0]` 등과 비슷한 불릿 수·길이인지 확인.

3. **코드명 노출 검사**: 함수명·파일명·필드명이 CHANGELOG에 노출되면 제거.

4. **가이드 실제 화면 일치**: 새 탭/버튼 명칭이 코드와 일치하는지 샘플 확인.

5. **사이드바** (`guide/{ko,en}/_sidebar.md`): 새 파일 추가 시 반영.

## 작업 완료 후 보고 형식

사용자에게 다음 형식으로 요약 보고:

```markdown
## {버전} 릴리즈 문서 업데이트 완료

### CHANGELOG ({N}줄)
- Added: {항목수} / Changed: {항목수} / Fixed: {항목수}

### Guide ({M}개 파일)
- admin/: {파일 목록}
- workspace/: {파일 목록}
- 루트: {파일 목록}

### 포함하지 않은 내용 (판단 근거)
- ...

### 검토 포인트
- ...
```

## 주의사항

- **커밋하지 않음**: 작성 후 사용자 검토 대기. 커밋은 `/git_commit`으로 별도 진행.
- **기존 내용 삭제 금지**: 가이드는 추가·수정만.
- **한/영 동시 수정**: 한쪽만 바꾸지 말 것.
- **사이드바 업데이트**: 신규 가이드 파일이 생기면 `_sidebar.md` 반영.
- **이미지 플레이스홀더**: `<!-- 스크린샷: 설명 / 파일명: images/xxx.png -->`
- **세부 절차는 가이드로**: CHANGELOG에 명령어·설정값·단계 쓰지 말고 "자세한 설정은 가이드 참조" 톤 유지.
